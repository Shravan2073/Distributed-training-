PROJECT NAME
Distributed-training-

PURPOSE
This project is a small script-based prototype for distributed deep learning across multiple machines connected over a network, with Tailscale as the intended connectivity layer.

The main goal is to let one machine act as the master node and one or more other machines act as worker nodes, then train a PyTorch model using Distributed Data Parallel (DDP). The code also includes:
- an interactive launcher for easier setup
- a GPU/environment diagnostic script
- basic checkpoint saving on the master node

The repository is not a packaged library. It is a collection of standalone Python scripts meant to be run directly.


TOP-LEVEL FILES
1. launch.py
Interactive CLI launcher that collects user input, checks connectivity, detects GPUs, and then execs into train.py.

2. train.py
Core distributed training script. Handles argument parsing, process-group setup, dataset loading, model creation, training, synchronization, and checkpointing.

3. gpu_test.py
Standalone environment diagnostic script for NVIDIA, PyTorch CUDA, TensorFlow GPU visibility, cuDNN, and WSL detection.

4. requirements.txt
Pinned dependencies for PyTorch/TensorFlow/CIFAR tooling and some CUDA/NCCL-related packages.

5. .gitignore
Minimal git ignore file.


HIGH-LEVEL SYSTEM ARCHITECTURE
The system is organized as three layers:

1. User-facing orchestration layer
- launch.py
- Purpose: guide the user through startup and reduce configuration mistakes.

2. Distributed training engine
- train.py
- Purpose: initialize PyTorch distributed communication and run the training loop.

3. Environment validation / troubleshooting
- gpu_test.py
- Purpose: help the user verify whether GPU and software prerequisites are working.


RUNTIME ARCHITECTURE
Typical runtime flow:

1. A user runs `python launch.py` on each machine.
2. launch.py detects whether the machine is the master or a worker.
3. launch.py checks GPU availability and chooses a backend:
- `nccl` for CUDA-capable machines
- `gloo` for CPU fallback
4. launch.py gathers:
- rank
- world size
- master address
- master port
- epochs
- batch size
5. launch.py replaces itself with `train.py` using `os.execv(...)`.
6. train.py initializes a PyTorch distributed process group.
7. Every process loads the CIFAR-10 training dataset.
8. A `DistributedSampler` partitions the dataset across ranks.
9. Each process creates the same model and wraps it in DDP.
10. Each process trains on its own data shard.
11. DDP automatically synchronizes gradients across all processes.
12. A barrier is reached after each epoch.
13. Rank 0 logs epoch-level progress and saves checkpoints.
14. At the end, the distributed process group is destroyed.


NETWORK MODEL
The project assumes a multi-machine environment.

Key concepts:
- Rank 0 is the master process.
- Every machine runs one process.
- `world_size` is the total number of participating processes/machines.
- Workers connect to the master using `MASTER_ADDR` and `MASTER_PORT`.
- Tailscale is the intended network transport, but the code itself uses standard socket-level addressing and PyTorch distributed backends.

Important implication:
- This code is oriented around one process per machine, not a multi-process-per-GPU local launcher like `torchrun`.


DISTRIBUTED COMMUNICATION MODEL
PyTorch distributed is used through:
- `torch.distributed`
- `DistributedDataParallel`
- `DistributedSampler`

Initialization modes supported in code:
1. `env://`
- Default mode.
- Sets `MASTER_ADDR` and `MASTER_PORT` environment variables.

2. `tcp://`
- Code path exists, but the implementation is incomplete/confusing.
- The script sets `MASTER_ADDR` and `MASTER_PORT`, then passes `init_method="tcp://"` directly.
- In normal PyTorch usage, a full URL like `tcp://host:port` is usually expected.
- This is a noteworthy design issue for any future refactor.

3. `file://<path>`
- Uses a file-based rendezvous store.
- Ensures the parent directory exists before initialization.


DEVICE / BACKEND MODEL
Backend selection behavior:

From launch.py:
- If CUDA is available, backend defaults to `nccl`.
- Otherwise, user is prompted whether to continue on CPU.
- CPU fallback uses `gloo`.

From train.py:
- If backend is `nccl` and CUDA is available, a CUDA device is selected.
- The device index is computed as `rank % torch.cuda.device_count()`.
- If `nccl` is requested but CUDA is not available, the script warns and downgrades to CPU with `gloo`.

Important implication:
- The device assignment logic is simple and assumes ranks can be mapped modulo local GPU count.
- In a true multi-machine environment with one process per node, this is usually acceptable if each node has one GPU.
- It is not a sophisticated topology-aware device placement strategy.


DATA PIPELINE
The actual training data pipeline is in train.py and uses CIFAR-10.

Steps:
1. Build transform:
- `transforms.ToTensor()`
- `transforms.Normalize(...)` using CIFAR-10 mean/std

2. Load dataset:
- `datasets.CIFAR10(root=args.data_dir, train=True, download=True, transform=transform)`

3. Split across ranks:
- `DistributedSampler(dataset, num_replicas=args.world_size, rank=args.rank, shuffle=True)`

4. Feed batches:
- `DataLoader(..., batch_size=args.batch_size, sampler=sampler, num_workers=2, pin_memory=(device.type == "cuda"))`

Implications:
- The dataset is downloaded independently by each process unless already present.
- There is no explicit validation dataset.
- There is no evaluation loop.
- There is no resume-from-checkpoint path wired into main training flow.


MODEL ARCHITECTURE
The model is `SimpleNet`, a very small classifier.

Actual structure:
1. Flatten input
2. Linear layer from `32 * 32 * 3` to `256`
3. ReLU
4. Dropout(0.2)
5. Linear layer `256 -> 128`
6. ReLU
7. Linear layer `128 -> 10`

Notes:
- Despite the class docstring saying "Small CNN for CIFAR-10", this is not a CNN.
- It is a fully connected multilayer perceptron over flattened image input.
- The model is intentionally simple and likely intended as a starter example rather than a high-accuracy baseline.


TRAINING PIPELINE
Per-run pipeline in train.py:

1. Parse CLI args
2. Validate `world_size` and `rank`
3. Determine device/backend compatibility
4. Initialize process group
5. Build transforms and load CIFAR-10
6. Create `DistributedSampler`
7. Create `DataLoader`
8. Build model
9. Wrap model with `DistributedDataParallel`
10. Create:
- loss function: `CrossEntropyLoss`
- optimizer: `Adam`
- scheduler: `StepLR(step_size=2, gamma=0.5)`
11. For each epoch:
- call `train_one_epoch(...)`
- call `scheduler.step()`
- call `dist.barrier()`
- rank 0 prints epoch summary
- rank 0 saves checkpoint
12. Print completion message on rank 0
13. Cleanup process group in `finally`


CHECKPOINT PIPELINE
Checkpoint saving is master-only.

Behavior:
- Only rank 0 writes checkpoint files.
- Directory is created if missing.
- File format: `checkpoint_epoch{epoch}.pt`
- Saved fields:
  - `epoch`
  - `model` from `model.module.state_dict()`
  - `optimizer` state
  - `loss`

Load support:
- `load_checkpoint(...)` exists as a helper.
- It loads model and optimizer state and returns the saved epoch.
- This function is currently not integrated into `main()`.

Implication:
- Resume support exists only as partial plumbing, not as a complete feature.


ERROR HANDLING / RESILIENCE DESIGN
The project includes several resilience features:

1. Worker retry logic during distributed initialization
- Non-master ranks can retry transient initialization failures.
- Transient errors include:
  - connection refused
  - timed out
  - failed to recv, got 0 bytes
  - connection reset
  - connection closed
  - broken pipe

2. Human-readable network diagnostics in launch.py
- DNS resolution checks
- port reachability checks
- firewall guidance
- Tailscale troubleshooting hints

3. Barrier error handling
- If synchronization fails after an epoch, the script prints that one or more workers may have crashed and aborts.

4. Cleanup in `finally`
- Attempts to destroy the process group even when training fails.


KNOWN OPERATIONAL ASSUMPTIONS
The project currently assumes:
- every machine can reach the master over the configured IP/port
- all processes use the same `world_size`
- each process has identical code and compatible dependencies
- all participants use a consistent backend
- dataset download is possible
- the master starts before workers


FUNCTION-BY-FUNCTION REFERENCE

FILE: launch.py

1. `detect_cuda_gpus()`
Purpose:
- Checks whether PyTorch is installed.
- Checks whether CUDA is available.
- If available, gathers GPU names.

Returns:
- `(False, "PyTorch not installed")` if torch import fails
- `(False, "CUDA not available")` if CUDA is unavailable
- `(True, message)` with detected GPU names otherwise

Role in system:
- Backend selection helper for the launcher.


2. `check_tailscale()`
Purpose:
- Runs `tailscale ip -4` in a subprocess.
- Reads IPv4 output.

Returns:
- detected Tailscale IPv4 address as a string
- `None` if Tailscale is missing, not running, or not returning an IP

Role in system:
- Helps auto-fill the master address or local node address.


3. `check_dns_resolution(host)`
Purpose:
- Verifies whether a hostname/IP can be resolved by `socket.gethostbyname`.

Returns:
- `True` on success
- `False` on failure

Side effect:
- Prints a resolution error message on failure.

Role in system:
- Used by troubleshooting logic.


4. `check_port_open(host, port, timeout=3)`
Purpose:
- Attempts to open a TCP connection to a host/port.

Returns:
- `(True, None)` on success
- `(False, error_message)` on failure

Failure cases handled:
- socket timeout
- DNS errors
- connection refused
- permission error
- generic OS/network error

Role in system:
- Worker pre-flight connectivity test.


5. `diagnose_connection_issue(host, port)`
Purpose:
- Prints a structured troubleshooting sequence for failed worker-to-master connections.

What it does:
- checks DNS resolution
- checks port reachability
- prints OS-specific firewall/network suggestions
- prints Tailscale-specific suggestions
- prints master-process verification hints

Role in system:
- Human support path, not part of training logic itself.


6. `main()`
Purpose:
- Orchestrates the interactive launcher flow.

Detailed flow:
- prints banner
- detects Tailscale IP or asks user to enter IP manually
- asks whether the user is master or worker
- asks for world size, port, epochs, batch size
- runs GPU detection and chooses backend
- if master:
  - rank = 0
  - master address = local IP
- if worker:
  - asks for worker rank
  - asks for master address
  - runs connectivity pre-flight
  - optionally aborts if the master is not reachable
- builds the `train.py` command line
- replaces current process using `os.execv`
- handles launcher-level file/permission/general exceptions

Role in system:
- Main user entry point.


FILE: train.py

1. `parse_args()`
Purpose:
- Defines and parses all CLI arguments for distributed training.

Arguments:
- `--rank`
- `--world-size`
- `--master-addr`
- `--master-port`
- `--backend`
- `--epochs`
- `--batch-size`
- `--lr`
- `--data-dir`
- `--save-dir`
- `--log-interval`
- `--ifname`
- `--init-timeout-min`
- `--init-retries`
- `--retry-delay`
- `--init-method`

Role in system:
- Configuration entry point for train.py.


2. `setup(rank, world_size, master_addr, master_port, backend, timeout_min, init_retries, retry_delay, ifname, init_method)`
Purpose:
- Configures and initializes the PyTorch distributed process group.

Behavior:
- supports `file://`
- supports `tcp://`
- defaults to `env://`
- sets environment variables for master address/port as needed
- optionally sets:
  - `GLOO_SOCKET_IFNAME`
  - `NCCL_SOCKET_IFNAME`
  - `TP_SOCKET_IFNAME`
- computes retry attempts
- computes timeout
- loops through initialization attempts
- prints informative status messages
- classifies several initialization errors into more actionable messages
- retries transient worker failures

Role in system:
- Core distributed rendezvous and networking setup.


3. `cleanup()`
Purpose:
- Calls `dist.destroy_process_group()`.

Role in system:
- Cleanup hook after training or failure.


4. `SimpleNet(nn.Module)`
Purpose:
- Defines the trainable model.

Methods:
- `__init__()`: constructs the sequential linear network
- `forward(x)`: runs the tensor through `self.net`

Role in system:
- Example CIFAR-10 classifier used by the training pipeline.


5. `save_checkpoint(model, optimizer, epoch, loss, save_dir, rank)`
Purpose:
- Saves training state to disk on rank 0 only.

Behavior:
- returns immediately for nonzero ranks
- creates save directory
- saves checkpoint dictionary via `torch.save`

Role in system:
- Persist training progress.


6. `load_checkpoint(model, optimizer, path, device)`
Purpose:
- Loads model and optimizer state from a checkpoint file.

Behavior:
- uses `torch.load(..., map_location=device)`
- restores `model.module`
- restores optimizer state
- prints resume message
- returns stored epoch

Role in system:
- Intended resume helper, currently unused in main flow.


7. `train_one_epoch(model, loader, sampler, optimizer, loss_fn, device, epoch, rank, log_interval)`
Purpose:
- Runs one epoch of local training for the current rank.

Detailed flow:
- sets model to train mode
- updates sampler epoch for distributed shuffling consistency
- initializes metrics
- creates rank-specific tqdm progress bar
- loops through batches
- moves data/targets to device
- zeroes gradients
- runs forward pass
- computes loss
- backpropagates
- updates optimizer
- accumulates total loss and accuracy statistics
- updates progress bar postfix every `log_interval` batches or on last batch
- returns average loss and accuracy

Role in system:
- Core per-epoch compute routine.


8. `main()`
Purpose:
- Orchestrates the full distributed training program.

Detailed flow:
- parse args
- validate world size
- validate rank range
- select device and backend fallback behavior
- print chosen device
- call setup(...)
- on setup failure: print fatal message and exit
- create CIFAR-10 transform
- load dataset
- create distributed sampler
- create dataloader
- instantiate model
- wrap with DDP
- create loss/optimizer/scheduler
- if rank 0, print run summary
- loop over epochs:
  - call `train_one_epoch`
  - step scheduler
  - barrier
  - if rank 0:
    - print epoch summary
    - save checkpoint
- if rank 0, print "Training complete!"
- handle keyboard interrupt
- print traceback for training exceptions
- always attempt cleanup

Role in system:
- Main distributed training entry point.


FILE: gpu_test.py

This file is script-style and mostly top-level code, not function-based except for one helper.

1. `separator(title)`
Purpose:
- Prints a formatted section header.

Role in system:
- Cosmetic helper for diagnostics output.


Top-level diagnostic sections in execution order:

2. NVIDIA-SMI section
Purpose:
- Executes `nvidia-smi`
- prints output if available
- reports if missing or failing


3. PyTorch GPU Check section
Purpose:
- imports torch
- prints torch version
- prints CUDA availability
- if CUDA is available:
  - prints CUDA version
  - prints GPU count
  - prints each GPU name
  - prints total VRAM
  - prints currently allocated/reserved memory
  - runs a quick GPU matrix multiplication test
- otherwise prints CPU fallback note


4. TensorFlow GPU Check section
Purpose:
- imports tensorflow
- prints tensorflow version
- lists physical GPU devices
- reports if no GPU is visible
- detects one specific NumPy ABI mismatch pattern and prints a suggested fix


5. CUDA / cuDNN Info section
Purpose:
- prints whether cuDNN is enabled in torch
- prints cuDNN version when CUDA is available


6. WSL Environment Check section
Purpose:
- reads `/proc/version`
- detects whether the runtime appears to be WSL


7. SUMMARY section
Purpose:
- summarizes whether PyTorch sees an active GPU
- prints some suggested remediation steps when GPU support is unavailable

IMPORTANT DESIGN NOTES AND INCONSISTENCIES
These are important for any LLM or engineer reading the code:

1. Documentation mismatch: MNIST vs CIFAR-10
- train.py top docstring and argument parser description mention MNIST.
- Actual dataset code uses CIFAR-10.

2. Documentation mismatch: CNN vs MLP
- `SimpleNet` docstring says "Small CNN for CIFAR-10".
- The model is not a CNN.
- It is a fully connected MLP over flattened images.

3. Encoding artifacts
- Some printed characters appear garbled in source text, likely from encoding issues.
- These do not appear to change core logic but affect readability.

4. TensorFlow is diagnostic-only
- TensorFlow is included in requirements and gpu_test.py.
- The training pipeline does not use TensorFlow at all.

5. Resume helper is incomplete as a feature
- `load_checkpoint(...)` exists but is not wired into `main()`.

6. `tcp://` init handling is questionable
- The code path likely does not provide a full TCP init URL as PyTorch commonly expects.
- This may need correction if that mode is actually used.

7. No evaluation/validation stage
- Training is implemented.
- Validation/testing metrics are not.

8. No test suite
- There are no unit tests, integration tests, or smoke-test scripts beyond manual runtime scripts.

9. No package/module abstraction
- Everything is script-oriented.
- Reuse and extension will likely require refactoring if the project grows.

BOTTOM-LINE SUMMARY
This project is a script-based distributed PyTorch training demo for multi-machine training over a network, intended to be easy to launch with Tailscale. The real heart of the system is `train.py`, while `launch.py` improves usability and `gpu_test.py` helps diagnose environment problems. The code is functional as an experimental scaffold, but it still has several inconsistencies and unfinished edges that should be understood before extending it.
